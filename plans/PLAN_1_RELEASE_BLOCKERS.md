## Phase 1.1 — Wire the real release signing key into the Android build

**Problem:** `android/app/build.gradle` only defines one `signingConfig` — the
well-known Android debug key (`debug.keystore` / `storePassword 'android'` /
`keyAlias 'androiddebugkey'`) — and the `release` build type reuses it
(`android/app/build.gradle:104-119`, specifically line 119
`signingConfig signingConfigs.debug` inside `buildTypes.release`). The real
production keystore already exists at `~/secrets/almahajjah/riwaq-release.jks`
(moved there deliberately in an earlier session — see `GLITCH_LOG.md` item 5)
but nothing references it. Any APK/AAB built locally right now is signed with
the public debug key, which Play Console will reject (or worse, silently
accept as a permanent wrong signing identity if it's ever the first upload).

**Note:** `android/` is gitignored and regenerated by `expo prebuild`, so this
fix must be applied in a way that survives a clean prebuild — either via an
Expo config plugin / `eas.json` credentials, or via a checked-in gradle
snippet that expo prebuild preserves. Check which approach this project uses
before editing.

**Fix:**
1. Determine whether this project ships via local `gradlew assembleRelease`
   or via `eas build --profile production` (check `eas.json`). If EAS: this is
   likely already solved by `eas credentials` managing the keystore in the
   cloud — verify by running `eas credentials` and checking if a production
   Android keystore is already configured there. If it is, this phase may
   just need verification, not new code.
2. If local gradle builds are the real release path: add a gitignored
   `android/keystore.properties` (storeFile, storePassword, keyAlias,
   keyPassword pointing at `~/secrets/almahajjah/riwaq-release.jks`) and add a
   `signingConfigs.release` block in `android/app/build.gradle` that reads
   from it, then point `buildTypes.release.signingConfig` at
   `signingConfigs.release` instead of `signingConfigs.debug`. Since
   `android/` isn't committed, encode this via an Expo config plugin (e.g. a
   `withAndroidSigningConfig` mod in a local plugin file referenced from
   `app.json`'s `plugins` array) so it's regenerated correctly on every
   `expo prebuild`.
3. Verify with `keytool -list -v -keystore <path-to-riwaq-release.jks>` vs.
   the built APK's signature (`apksigner verify --print-certs app-release.apk`)
   — the SHA-256 fingerprints must match the real keystore, not the debug one.

**Do not** ask the owner for the keystore password over chat/plaintext if
avoidable — read it from wherever it's already stored securely (owner will
know where), or ask them to run the credential-setup step themselves if it
requires typing a password interactively.

## Phase 1.2 — Kill the live demo-admin backdoor

**Problem (confirmed live, not theoretical):** the seeded demo account
`admin@gmail.com` with password `test55%%` is still active in the
**production** Supabase auth (project `prpyxnxgkpspjoxvcaro`) with
`profiles.role = 'admin'`, and this exact password still authenticates it —
verified live with a real login call during this test. Anyone who reads
`scripts/seed-auth.mjs` (a plain file in this repo) and knows/guesses that
password gets **full admin control**: publish/unpublish any content, read
every user's email and progress, ban accounts.

`user@gmail.com`, `publisher@gmail.com`, and `sheikh@gmail.com` also still
exist in production but no longer authenticate with `test55%%` (already
rotated) — they should still be removed as dead weight.

Separately, `src/config.ts:42-45` (`DEMO_ACCOUNTS` — same email/password
pair) ships unconditionally inside the production JS bundle (only the
sign-in *bypass logic* in `src/api/auth.ts` is gated by `USE_MOCK`, not the
credential strings themselves) — so even after the live account is
neutralized, the literal string `admin@gmail.com` / `test55%%` is
recoverable by unzipping the shipped APK and grepping the JS bundle.

**Fix — two independent halves, do both:**

1. **Production data (the account itself):** the owner previously said
   ("PLAN_SECURITY.md closeout") they wanted to handle this personally later
   — check with them before deleting/rotating, since it's live production
   auth data outside normal code-review scope. Recommended action once
   confirmed: `DELETE /auth/v1/admin/users/{id}` for all four
   `*@gmail.com` demo accounts (find ids via
   `GET /auth/v1/admin/users?filter=<email>`), or at minimum rotate
   `admin@gmail.com` to a strong random password. This is a live Supabase
   Auth Admin API call, not a code change — do not attempt without the
   owner's explicit go-ahead in this session, since it mutates production
   user data outside version control.

2. **Code (stop shipping the literal credentials):** in `src/config.ts`,
   stop exporting `DEMO_ACCOUNTS` unconditionally. Either:
   - Gate the export itself behind `USE_MOCK` (e.g.
     `export const DEMO_ACCOUNTS = USE_MOCK ? {...} : undefined`, with
     call-sites in `src/api/auth.ts` already checking `USE_MOCK` before use,
     so this should be a no-op behaviorally but removes the strings from the
     release Hermes bundle since dead-code elimination can drop the unused
     branch), or
   - Move the object to a dev-only module that's excluded from Metro's
     release bundle entirely (e.g. `src/mock/demoAccounts.ts`, imported only
     from `src/mock/api.ts` which is presumably already mock-only).
   Verify the fix by building a release bundle and grepping the output JS for
   `test55` — it must not appear.

**Suggested follow-up (not blocking, worth a mention to the owner):** a
pre-release CI check or script that fails the build if any of the four known
demo emails still resolve in the production project, so this can't silently
regress on a future release.
